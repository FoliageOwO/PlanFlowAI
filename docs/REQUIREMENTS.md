## 1. 引言

### 1.1 目的

本文档定义 PlanFlow AI 系统的完整需求规格。系统旨在解决用户从多渠道接收零散信息后需要手动整理任务、设置提醒的痛点，通过 OCR、文档解析和大语言模型实现从非结构化信息到结构化任务计划的自动化转换。

### 1.2 适用范围

本文档面向项目开发团队、课程评审老师和答辩评审，作为系统开发、测试和验收的依据。

### 1.3 定义与缩写

| 术语 | 说明 |
|------|------|
| **OCR** | 光学字符识别，从图片中提取文字 |
| **AI 解析** | 使用大语言模型对文本进行语义理解和结构化提取 |
| **source_input** | 用户输入的原始信息（文本/文件） |
| **parse_job** | 后台解析任务，处理 source_input |
| **raw_text** | OCR/文档提取后的纯文本 |
| **提醒渠道** | 通知触达方式（站内、本地、浏览器、邮件） |
| **MVP** | 最小可行产品 |
| **SSE** | Server-Sent Events，服务端推送 |
| **JWT** | JSON Web Token，鉴权令牌 |

## 2. 项目概述

### 2.1 项目背景

学生、上班族和普通用户经常从不同渠道接收大量零散信息，例如课程群公告、老师布置的作业要求、项目截止通知、会议安排、活动海报、聊天记录、PDF 通知、截图等。这些信息通常是非结构化的，用户需要手动阅读、理解、拆任务、记时间、设置提醒，容易遗漏截止时间、提交要求和准备事项。

传统待办软件要求用户手动输入任务标题、截止时间、备注和提醒规则，信息录入成本高。PlanFlow AI 通过 OCR、文档解析和大语言模型，把杂乱信息自动转化成可执行计划，降低用户整理任务和安排时间的成本。

### 2.2 项目目标

实现一个 **Web + Android 双端可用** 的 AI 任务规划系统：

- **Web 端**：React 部署到公网域名，便于教师、同学和普通用户直接访问
- **Android 端**：Capacitor 打包 APK，复用同一套 React 前端代码，支持本地定时提醒
- **后端**：Spring Boot + MySQL，负责用户管理、文件上传、异步任务队列、OCR 调用、AI 解析、任务生成

### 2.3 核心价值

1. 从非结构化信息中自动识别目标
2. 将大目标拆分为可执行任务
3. 自动识别绝对时间和相对时间
4. 提取地点、文件格式、提交方式、命名格式等约束
5. 自动生成检查清单
6. 自动生成时间轴
7. 自动生成提醒规则
8. 根据截止时间和任务优先级生成规划建议
9. 多来源信息合并进统一任务系统
10. Android APK 端使用本地通知提醒用户

## 3. 用户角色

### 3.1 角色定义

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| **学生** | 课程学习者，需要整理课程通知、作业、考试安排 | 快速录入课程信息、自动生成待办、截止提醒 |
| **上班族** | 职场人员，需要管理会议、项目节点、文档提交 | 多源信息整合、时间轴管理、优先级排序 |
| **普通用户** | 生活场景用户，需要管理预约、活动、账单 | 简单操作、直观展示、及时提醒 |

> **第一版重点服务学生场景**，例如课程项目立项、实验报告提交、考试复习安排、答辩准备。

### 3.2 用户特征

- 基本计算机操作能力
- 可能不具备技术背景
- 需要在手机和电脑间切换使用
- 希望减少手动录入工作

## 4. 功能性需求

### 4.1 认证与用户管理 (AUTH)

| 编号 | 需求名称 | 描述 | 优先级 |
|------|----------|------|--------|
| AUTH-01 | 用户注册 | 用户通过用户名、密码、昵称注册新账号 | P0 |
| AUTH-02 | 用户登录 | 用户通过用户名和密码登录，获取 JWT 令牌 | P0 |
| AUTH-03 | 获取当前用户 | 登录状态下获取当前用户信息 | P0 |
| AUTH-04 | 修改个人信息 | 修改昵称、邮箱等个人信息 | P1 |
| AUTH-05 | 密码安全 | 密码使用 BCrypt 加密存储，不返回明文密码 | P0 |

### 4.2 输入源管理 (INPUT)

| 编号 | 需求名称 | 描述 | 优先级 |
|------|----------|------|--------|
| INPUT-01 | 自然语言输入 | 用户输入一句话或公告文本，系统创建解析任务 | P0 |
| INPUT-02 | 图片上传 | 用户上传 jpg/png/webp 图片，系统保存并创建解析任务 | P0 |
| INPUT-03 | PDF 文件上传 | 用户上传 PDF 文件，系统提取文本并创建解析任务 | P0 |
| INPUT-04 | Word 文件上传 | 用户上传 docx 文件，系统提取文本并创建解析任务 | P0 |
| INPUT-05 | 输入源列表 | 查看用户所有历史输入源 | P1 |
| INPUT-06 | 输入源详情 | 查看输入源原文、OCR 文本、解析状态、解析结果 | P1 |
| INPUT-07 | 删除输入源 | 删除输入源及相关解析结果 | P1 |
| INPUT-08 | 文件类型校验 | 只允许上传指定类型文件，拒绝非法扩展名 | P0 |
| INPUT-09 | 文件大小限制 | 单文件最大 20MB | P0 |

### 4.3 文档解析 (EXTRACT)

| 编号 | 需求名称 | 描述 | 优先级 |
|------|----------|------|--------|
| EXTRACT-01 | PDF 文本提取 | 使用 PDFBox 提取可复制文本 PDF 的内容 | P0 |
| EXTRACT-02 | DOCX 文本提取 | 使用 Apache POI 提取 Word 文档段落文本 | P0 |
| EXTRACT-03 | 文本清洗 | 去除多余空白、合并断行、保留段落结构 | P0 |
| EXTRACT-04 | 长度限制 | 限制输入给 AI 的文本最大长度（如 8000 字符），超长截断 | P0 |
| EXTRACT-05 | 图片型 PDF 检测 | 检测到不可提取文本的 PDF 时提示用户 | P1 |

### 4.4 OCR 识别 (OCR)

| 编号 | 需求名称 | 描述 | 优先级 |
|------|----------|------|--------|
| OCR-01 | 图片文字识别 | 调用 PaddleOCR 服务识别图片中的文字 | P0 |
| OCR-02 | 返回识别文本 | 返回完整的识别文本 | P0 |
| OCR-03 | 返回置信度 | 返回每行文字的识别置信度（可选） | P1 |
| OCR-04 | 前端展示原文 | 展示 OCR 识别出的原文，允许用户编辑后重新解析 | P1 |
| OCR-05 | OCR 服务部署 | PaddleOCR 封装为独立 FastAPI 服务，与 Spring Boot 分离部署 | P0 |

### 4.5 AI 语义解析 (AI)

| 编号 | 需求名称 | 描述 | 优先级 |
|------|----------|------|--------|
| AI-01 | 信息抽取 | 从原始文本中提取目标、任务、时间、地点、约束 | P0 |
| AI-02 | 任务拆解 | 将大目标拆分为可执行的具体任务 | P0 |
| AI-03 | 时间标准化 | 将相对时间（如"下周五"）转换为绝对时间 | P0 |
| AI-04 | 检查清单生成 | 自动生成可核对的任务检查项 | P0 |
| AI-05 | 提醒建议生成 | 自动建议提醒时间 | P0 |
| AI-06 | 风险提示 | 识别可能的风险和冲突 | P1 |
| AI-07 | 规划建议 | 根据截止时间和优先级给出规划建议 | P1 |
| AI-08 | 结构化输出 | 模型必须输出严格 JSON，后端校验后落库 | P0 |
| AI-09 | 失败重试 | JSON 解析失败时自动重试一次 | P0 |
| AI-10 | 模型可替换 | 模型调用封装为统一接口，支持 DeepSeek / Qwen 切换 | P1 |

## 5. 非功能性需求

### 5.1 性能

| 编号 | 需求名称 | 目标值 |
|------|----------|--------|
| NFR-01 | 文本解析响应 | 用户提交文本后，前端应在 5 秒内收到 jobId，完整解析应在 30 秒内完成 |
| NFR-02 | 图片 OCR + 解析 | 完整 OCR → AI 解析流程应在 60 秒内完成 |
| NFR-03 | 任务列表查询 | 列表查询应在 1 秒内返回 |
| NFR-04 | 并发用户 | 支持至少 20 个用户同时使用 |
| NFR-05 | 文件上传 | 单文件最大 20MB，上传应在 30 秒内完成 |

### 5.2 安全

| 编号 | 需求名称 | 要求 |
|------|----------|------|
| NFR-06 | 密码存储 | 使用 BCrypt 加密，不存储明文 |
| NFR-07 | 鉴权 | 除登录注册外所有接口需要 JWT 验证 |
| NFR-08 | 数据隔离 | 用户只能访问自己的数据 |
| NFR-09 | 文件安全 | 文件类型白名单校验，防止路径穿越 |
| NFR-10 | 密钥管理 | AI API Key 仅存于后端环境变量，不泄露到前端 |
| NFR-11 | 文件路径 | 不上传文件绝对路径到前端，使用相对路径或 ID 引用 |

### 5.3 可用性

| 编号 | 需求名称 | 要求 |
|------|----------|------|
| NFR-12 | 错误提示 | AI 解析失败时给出明确、友好的错误提示 |
| NFR-13 | OCR 失败处理 | OCR 失败时允许用户手动编辑识别文本后重新解析 |
| NFR-14 | 自动重试 | JSON 解析失败时自动重试一次 |
| NFR-15 | 无时间任务 | 无明确时间的任务仍可保存，不生成默认提醒 |
| NFR-16 | 权限拒绝处理 | Android 通知权限被拒绝时，提示用户可在设置中开启 |
| NFR-17 | 空状态 | 列表为空时展示友好提示和操作引导 |

### 5.4 可维护性

| 编号 | 需求名称 | 要求 |
|------|----------|------|
| NFR-18 | 模块化 | OCR、AI、任务生成分别封装为独立服务类 |
| NFR-19 | 可替换性 | AI 模型供应商可替换，通过统一接口实现 |
| NFR-20 | 可升级队列 | 任务队列可从 MySQL 升级到 Redis/RabbitMQ，抽象队列接口 |
| NFR-21 | 可扩展提醒 | 提醒渠道通过插件机制扩展 |
| NFR-22 | 代码规范 | 遵循 Java 和 TypeScript 编码规范，使用统一代码风格 |
| NFR-23 | 日志 | 关键操作和异常记录日志 |

### 5.5 兼容性

| 编号 | 需求名称 | 要求 |
|------|----------|------|
| NFR-24 | 浏览器 | 支持 Chrome 90+、Edge 90+、Safari 14+、Firefox 90+ |
| NFR-25 | Android | 支持 Android 12+ (API 31+) |

## 8. 数据需求

### 8.1 数据库

- **数据库系统**：MySQL 8.0+
- **命名规则**：snake_case 表名，bigint 主键，datetime 时间字段
- **核心表**：user, user_setting, source_input, parse_job, ai_analysis_result, task, task_checklist_item, timeline_event, reminder_rule, notification, device_sync_record

### 8.2 文件存储

- **存储方式**：本地磁盘 uploads/ 目录
- **路径规则**：`uploads/{userId}/{sourceInputId}/{原文件名}`
- **文件类型**：jpg, jpeg, png, webp, pdf, docx

## 9. 约束与假设

### 9.1 技术约束

| 约束 | 说明 |
|------|------|
| 前端框架 | 必须使用 React + TypeScript + Vite |
| 后端框架 | 必须使用 Spring Boot + Java 17 |

| 移动端 | 必须使用 Capacitor 打包 APK（非原生开发） |

| AI 模型 | 使用 DeepSeek 或 Qwen API（国产模型） |

| 假设 | 说明 |
|------|------|
| 用户网络 | 用户具备稳定的互联网连接 |
| 中文场景 | 系统主要处理中文输入，AI 模型需支持中文 |
| 单用户 | 第一版不涉及多人协作场景 |
| 课程周期 | 项目开发周期约 7 天 |
| 演示环境 | 使用测试数据演示，不需要生产级可靠性 |

## 10. MVP 范围界定

### 10.1 MVP（第一版必须完成）

| 模块 | 包含功能 |
|------|----------|
| 用户认证 | 注册、登录、JWT、个人信息 |
| 输入源 | 文本输入、图片上传、PDF 上传、文件类型校验、大小限制 |
| 文档解析 | PDF 文本提取、DOCX 文本提取、文本清洗 |
| OCR | 图片文字识别（PaddleOCR） |
| AI 解析 | 信息抽取、任务拆解、时间标准化、检查清单生成、结构化 JSON 输出 |
| 异步队列 | 任务创建、状态跟踪、后台 Worker、失败记录 |
| 任务管理 | 列表、创建、编辑、状态流转、删除、详情 |
| 检查清单 | 增删改查、勾选 |
| 时间轴 | 时间线展示、按天分组、跳转任务 |
| 提醒规则 | AI 建议、默认规则、手动添加、取消 |
| 站内通知 | 列表、未读数量、标记已读、全部已读、跳转详情 |
| 设置 | 提醒偏好、通知开关、退出登录 |
| Android 端 | APK 打包、通知权限、本地通知注册、前台同步 |

### 10.2 增强（第一版不做）

| 功能 | 原因 |
|------|------|
| 多人协作 / 班级系统 | 超出课程项目范围 |
| 教师端 / 管理端 | 第一版仅面向个人用户 |
| 支付功能 | 不需要 |
| 应用商店上架 | 仅提供 APK 直接安装 |
| FCM / APNs 远程推送 | 使用本地通知替代 |
| 复杂权限系统 | 仅需 RBAC 基础模型 |
| 图片型 PDF 完整 OCR | 实现成本高，可提示用户 |
| 向量检索 / 语义搜索 | 第一版 LIKE 查询足够 |
| 多模型切换页面 | 后端封装接口，前端暂不暴露 |
| 浏览器通知 | API 复杂，优先级低 |
| 邮件通知 | 需要邮件服务器 |
| SSE 实时进度 | 第一版轮询足够 |

作为一位学生用户，

作为一位学生用户，

作为一位用户，
我希望系统能自动生成检查清单，
这样我就不会遗漏重要步骤。

作为一位安卓用户，
我希望到截止时间前能收到手机本地通知，
这样即使我忘记打开 App 也不会错过截止。

作为一位用户，
我希望所有输入来源的任务都在同一页面展示，
